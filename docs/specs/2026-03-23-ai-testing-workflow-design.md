# AI 自动化测试验收流程设计

## 1. 概述

本文档定义了 AI 辅助开发中的自动化测试验收流程，确保 AI 开发的每个功能都经过充分测试后才能标记为完成。该流程涵盖单元测试、E2E 测试、视觉回归测试和无障碍审计，通过自动化的方式在每次功能开发完成后执行完整测试验证。

## 2. 背景与目标

### 2.1 问题描述

当前开发过程中存在以下问题：
- AI 完成功能开发后缺少测试验证
- 同一问题反复修改仍不正确，缺乏自动化检测手段
- UI 视觉问题难以被发现，往往到用户反馈才发现
- 缺少标准化的测试流程，测试行为不规范

### 2.2 目标

建立标准化的 AI 测试验收流程，实现以下目标：
- 功能开发完成后自动执行完整测试
- 通过多重检测机制发现逻辑错误、交互问题、视觉偏差和无障碍问题
- 生成可视化的测试报告，包含测试结果和失败时的截图对比
- 通过自动重试机制减少人工干预，提升开发效率

## 3. 技术架构

### 3.1 测试框架选型

| 测试类型 | 框架 | 用途 |
|---------|------|------|
| 单元测试 | Vitest | 测试工具函数核心逻辑（json.ts、timestamp.ts 等） |
| E2E 测试 | Playwright | 模拟用户操作，验证页面交互流程 |
| 视觉回归 | Playwright Screenshot | 截图对比，检测 UI 布局、颜色、组件位置变化 |
| 无障碍审计 | axe-core | 检测可访问性问题（如缺少 alt 文本、对比度不足等） |

### 3.2 技术栈

- **测试运行器**: Vitest
- **浏览器自动化**: Playwright (Chromium, Firefox, WebKit)
- **截图对比**: Playwright 内置截图功能 + pixelmatch
- **无障碍检测**: axe-core + @axe-core/playwright
- **报告生成**: 自定义 HTML 报告生成器

## 4. 自动化流程

### 4.1 完整测试流程

```
功能开发完成
    ↓
启动开发服务器 (npm run dev)
    ↓
等待服务器就绪 (健康检查)
    ↓
运行单元测试 (Vitest)
    ↓
运行 E2E 测试 (Playwright)
    ↓
视觉回归测试 (截图对比)
    ↓
Console 错误检测
    ↓
无障碍审计 (axe-core)
    ↓
生成测试报告
    ↓
┌─────────────────────┐
│  测试通过           │──→ 任务完成，标记功能为已完成
└─────────────────────┘
    ↓
┌─────────────────────┐
│  测试失败           │──→ 自动重试 (最多 3 次)
└─────────────────────┘
    ↓
重试失败 → 报告详细错误（包含截图、日志、错误堆栈）
```

### 4.2 阶段详解

#### 阶段 1：启动开发服务器

- 执行 `npm run dev` 启动 Vite 开发服务器
- 等待服务器就绪（默认超时 60 秒）
- 验证服务器可访问（GET 请求根路径返回 200）

#### 阶段 2：单元测试

- 使用 Vitest 运行 `tests/*/unit/*.test.ts`
- 覆盖工具函数的正常路径和边界情况
- 测试失败立即停止后续测试

#### 阶段 3：E2E 测试

- 使用 Playwright 运行 `tests/*/e2e/*.spec.ts`
- 模拟真实用户在浏览器中的操作
- 验证功能正确性和交互流程
- 捕获测试过程中的 console 错误和网络失败

#### 阶段 4：视觉回归测试

- 对关键页面元素进行截图
- 与 baseline 目录中的基线截图对比
- 计算像素差异比例，超过阈值（默认 1%）则标记为失败
- 首次运行时自动创建基线截图

#### 阶段 5：Console 错误检测

- 监听页面加载和交互过程中的 console.error、console.warn
- 监听未捕获的 JavaScript 异常
- 监听网络请求失败（4xx、5xx 状态码）
- 任何错误都标记测试为失败

#### 阶段 6：无障碍审计

- 使用 axe-core 对关键页面进行审计
- 检测可访问性问题（缺少 alt 文本、对比度不足、键盘可聚焦等）
- 生成无障碍问题报告

#### 阶段 7：生成测试报告

- 终端输出测试摘要（通过/失败数、耗时）
- 生成 HTML 报告（tests/report.html）
- 失败时展示预期 vs 实际截图对比

### 4.3 自动重试机制

- 测试失败后自动重试，最多 3 次
- 每次重试前先尝试修复已知问题
- 重试间隔递增（5 秒、10 秒、15 秒）
- 3 次重试后仍失败，报告详细错误信息

## 5. 测试配置

### 5.1 基础配置 (config.ts)

```typescript
export const testConfig = {
  // 开发服务器配置
  devServer: {
    url: 'http://localhost:5173',
    timeout: 60000,
    healthCheckInterval: 1000,
  },

  // 浏览器配置
  browser: {
    headless: true,
    viewport: { width: 1280, height: 720 },
  },

  // 测试超时配置
  timeouts: {
    unit: 30000,
    e2e: 60000,
    visual: 30000,
    axe: 30000,
  },

  // 视觉对比阈值
  visual: {
    diffThreshold: 0.01, // 1% 像素差异
  },

  // 重试配置
  retry: {
    maxAttempts: 3,
    intervals: [5000, 10000, 15000],
  },
}
```

### 5.2 测试文件规范

每个功能的测试目录包含以下文件：

- **unit/*.test.ts**: 单元测试，使用 Vitest
- **e2e/*.spec.ts**: E2E 测试，使用 Playwright
- **visual/baseline/**: 基线截图目录
- **axe.config.ts**: 无障碍测试配置

## 7. 开发流程集成

### 7.1 AI 开发任务标准流程

每个 AI 开发任务必须包含以下阶段：

1. **需求分析**: 理解功能需求，评估实现复杂度
2. **设计**: 编写设计文档，获得用户批准
3. **实现**: 编写功能代码
4. **测试**: 执行自动化测试验收流程 ← 新增阶段
5. **提交**: 代码审查并提交

### 7.2 测试阶段强制要求

- 功能开发完成后，必须执行完整测试流程
- 测试必须通过（无失败项）才能标记任务为完成
- 测试失败时，必须进行自动重试（最多 3 次）
- 重试后仍失败，必须报告详细错误信息，包含截图和日志
- 禁止跳过测试或伪造测试结果

## 8. 最佳实践

### 8.1 单元测试最佳实践

- 测试函数应有明确的输入和预期输出
- 覆盖正常路径、边界情况和错误处理
- 使用描述性的测试名称，说明测试场景
- 保持测试独立，避免相互依赖

### 8.2 E2E 测试最佳实践

- 每个用户操作路径至少一个测试用例
- 使用真实用户场景，而非内部实现细节
- 测试前后清理测试数据
- 合理使用等待，避免 flaky 测试

### 8.3 视觉测试最佳实践

- 只测试关键的视觉元素，避免全页面截图
- 在一致的浏览器环境中运行（相同分辨率、字体、系统）
- 基线截图在功能稳定后创建
- 定期审查和更新基线截图

### 8.4 无障碍测试最佳实践

- 重点关注交互元素（按钮、输入框、链接）
- 确保键盘导航正常工作
- 检查颜色对比度是否符合 WCAG 标准
- 验证图片是否有替代文本

## 9. 未来扩展

- 支持 CI/CD 集成，在 Pull Request 时自动运行测试
- 支持测试覆盖率报告
- 支持性能测试（页面加载时间、交互响应时间）
- 支持跨浏览器测试（Chrome、Firefox、Safari）
- 支持移动端响应式测试

## 10. 术语表

| 术语 | 定义 |
|------|------|
| 单元测试 | 对代码中最小的可测试单元进行测试 |
| E2E 测试 | 端到端测试，模拟真实用户操作流程 |
| 视觉回归 | 通过截图对比检测 UI 变化 |
| 无障碍审计 | 检测应用是否符合可访问性标准 |
| 基线截图 | 正确状态的参考截图，用于对比 |
| flaky 测试 | 不稳定的测试，有时通过有时失败 |
