# DevTools 项目开发规则

> 所有开发人员（包括AI Agent）必须遵守本规则  
> 最后更新：2026-03-20

---

## 一、需求变更记录

### 1.1 变更记录要求

每次需求变更（新增、修改、删除功能）必须：

1. **记录变更内容**：在 `docs/changelog/` 目录下创建日期命名的变更记录
2. **更新产品文档**：同步更新 `docs/specs/` 下的产品设计文档版本号
3. **Git提交关联**：提交信息中引用变更记录文件

### 1.2 变更记录格式

文件命名：`YYYY-MM-DD-<简短描述>.md`

```markdown
# 变更记录 - YYYY-MM-DD

## 变更类型
- [ ] 新增功能
- [ ] 修改功能
- [ ] 删除功能
- [ ] Bug修复
- [ ] 重构

## 变更描述
简要说明本次变更的内容

## 变更原因
说明为什么需要这次变更

## 影响范围
- 影响的文件/模块：
- 是否需要更新文档：
- 是否需要更新测试：

## 相关文档
- 产品文档：docs/specs/2026-03-20-devtools-design.md (v1.x)
```

### 1.3 文档语言

文件名使用英文命名，文档内容主体使用中文，技术相关词语可使用英文

## 二、文档同步更新

### 2.1 文档更新时机

| 变更类型 | 必须更新的文档 |
|----------|----------------|
| 新增功能 | 产品设计文档、变更记录 |
| 修改功能规格 | 产品设计文档、变更记录 |
| 修改UI设计 | 产品设计文档 |
| 修改技术架构 | 产品设计文档 |
| 修复Bug | 变更记录（可选） |

### 2.2 版本号规范

产品设计文档版本号格式：`v主版本.次版本`

- **主版本**：重大架构变更或功能重构
- **次版本**：功能新增、修改或优化

更新规则：
- 新增功能 → 次版本 +1（v1.0 → v1.1）
- 重大变更 → 主版本 +1（v1.x → v2.0）

### 2.3 文档更新检查清单

每次代码改动前，检查：

- [ ] 本次改动是否涉及功能变更？
- [ ] 产品设计文档是否需要更新？
- [ ] 是否需要创建变更记录？
- [ ] 文档版本号是否需要递增？

## 三、Git 提交规范

### 3.1 提交信息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 3.2 Type 类型

| Type | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| docs | 文档更新 |
| style | 代码格式（不影响功能） |
| refactor | 重构 |
| test | 测试相关 |
| chore | 构建/工具相关 |

### 3.3 示例

```bash
# 新增功能
git commit -m "feat(json): 添加JSON转义功能

- 支持将JSON转为字符串转义格式
- 支持消除转义还原
- 更新产品文档至v1.2

相关变更: docs/changelog/2026-03-20-json-escape.md"

# 修复Bug
git commit -m "fix(timestamp): 修复毫秒时间戳识别错误

当时间戳为13位时自动识别为毫秒"

# 文档更新
git commit -m "docs: 更新产品文档至v1.1

- 添加主页面设计
- 添加工具卡片规范"
```

### 3.4 分支命名规范

| 分支类型 | 命名格式 | 示例 |
|----------|----------|------|
| 功能分支 | `feat/<功能名>` | `feat/json-formatter` |
| 修复分支 | `fix/<问题描述>` | `fix/timestamp-bug` |
| 文档分支 | `docs/<文档名>` | `docs/product-spec` |

## 四、代码风格

### 4.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | PascalCase（组件）/ kebab-case（其他） | `JsonFormatter.vue` / `json-utils.ts` |
| 组件名 | PascalCase | `JsonFormatter` |
| 变量/函数 | camelCase | `formatJson` / `isValidJson` |
| 常量 | UPPER_SNAKE_CASE | `MAX_INPUT_SIZE` |
| CSS类名 | kebab-case | `tool-card` / `json-output` |

### 4.2 注释要求

- **组件**：必须添加功能说明注释
- **复杂逻辑**：必须添加实现思路注释
- **工具函数**：必须添加 JSDoc 注释

```typescript
/**
 * 格式化JSON字符串
 * @param json - JSON字符串
 * @param indent - 缩进空格数，默认2
 * @returns 格式化后的JSON字符串
 * @throws {Error} JSON语法错误时抛出异常
 */
export function formatJson(json: string, indent: number = 2): string {
  // 实现逻辑
}
```

### 4.3 TypeScript 要求

- 所有函数参数和返回值必须有类型声明
- 避免使用 `any`，优先使用具体类型或泛型
- 使用 `interface` 定义对象结构

## 五、测试要求

### 5.1 测试覆盖范围

| 功能类型 | 测试要求 |
|----------|----------|
| 核心工具函数 | 必须有单元测试 |
| UI组件 | 建议有组件测试 |
| 边界情况 | 必须覆盖（空输入、超长输入、非法输入） |

### 5.2 测试文件组织

```
src/
├── utils/
│   ├── json.ts
│   └── json.test.ts      # 测试文件与源文件同目录
├── tools/
│   └── json-formatter/
│       ├── JsonFormatter.vue
│       └── JsonFormatter.test.ts
```

```typescript

  it('应该正确格式化有效的JSON', () => {
    // 测试逻辑
  });

  it('应该在JSON语法错误时抛出异常', () => {
    // 测试逻辑
  });

  it('应该支持自定义缩进', () => {
    // 测试逻辑
  });
});
```

### 5.4 完成检查清单

功能开发完成后，必须：

- [ ] 核心函数单元测试通过
- [ ] 边界情况测试覆盖
- [ ] 手动测试主要交互流程
- [ ] 无控制台错误/警告

## 六、开发流程

### 6.1 标准开发流程

```
1. 需求确认
   ↓
2. 更新产品文档（如有变更）
   ↓
3. 创建变更记录（如有变更）
   ↓
4. 创建功能分支
   ↓
5. 开发实现
   ↓
6. 编写测试
   ↓
7. 提交代码（遵循提交规范）
   ↓
8. 完成检查清单
```

### 6.2 AI Agent 工作规则

作为AI Agent，在每次开发任务中必须：

1. **任务开始前**：
   - 检查是否涉及需求变更
   - 如有变更，先更新产品文档
   - 创建变更记录文件

2. **开发过程中**：
   - 遵循代码风格规范
   - 添加必要注释
   - 编写测试用例

3. **任务完成后**：
   - 运行测试确认通过
   - 按规范提交代码
   - 更新相关文档版本号
   - **更新memory文件**（见6.3）

### 6.3 Memory文件维护规则

**文件位置：** `docs/memory/PROJECT_MEMORY.md`

**维护原则：**
- **精简优先**：只保留主要信息，删除冗余内容
- **可恢复性**：包含足够的信息以支持中断后快速恢复
- **时效性**：每次任务完成后立即更新

**必须更新的场景：**
| 场景 | 更新内容 |
|------|----------|
| 完成开发阶段 | 更新进度状态（⏳ → ✓） |
| 新增重要决策 | 记录到"关键决策记录" |
| 解决关键问题 | 记录到"问题解决记录" |
| 修改项目结构 | 更新"项目结构"部分 |
| 新增待办事项 | 更新"待办事项"部分 |

**更新内容规范：**
- **开发进度**：更新各阶段状态，只保留最近的5-10个Git提交
- **关键决策**：只记录影响架构或方向的决策，删除过时决策
- **问题解决**：只保留有参考价值的解决方案，删除已不相关的问题
- **项目结构**：保持与实际结构一致，删除不存在的文件说明
- **待办事项**：删除已完成项，优先级排序

**禁止内容：**
- 详细的代码实现（应放在代码注释中）
- 临时性的调试信息
- 重复的描述
- 过时的信息

**检查清单：**
- [ ] 开发进度是否准确？
- [ ] 是否有冗余信息需要删除？
- [ ] 关键决策是否已记录？
- [ ] 恢复指南是否仍然有效？

## 七、例外情况

以下情况可简化流程：

- **文档错别字修正**：无需创建变更记录
- **代码格式化**：无需更新产品文档
- **紧急修复**：可先修复后补充文档
